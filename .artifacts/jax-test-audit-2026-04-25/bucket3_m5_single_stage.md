# Bucket 3 M5 Single-Stage Objective Wrappers — Test Audit
Date: 2026-04-25 — Branch: gpu-purity-stage2-20260405

## Executive summary

1. The M5 wrappers (`BoozerResidualJAX`, `IotasJAX`, `NonQuasiSymmetricRatioJAX`) are exhaustively unit tested but the suite is **monkeypatch heavy**: the bulk of `test_surface_objectives_jax.py` (~3000 of 4322 lines) replaces every collaborator with a lambda, leaving "tests" that just plumb stubbed values through dict-shaped contracts. The CPU-vs-JAX integration coverage in `test_single_stage_jax_cpu_reference.py` is the real validation lane.
2. The well-conditioned-exact adjoint vs dense PLU oracle is missing here; `test_iotas_jax_exact_well_conditioned_gradient_matches_dense_projection` (line 1728) builds its own toy 3x3 matrix instead of solving an actual exact Boozer fixture, so the lane gate is implemented in `tests/geo/test_boozersurface_jax.py`, not in this bucket.
3. End-to-end FD gradient checks (TestBoozerResidualGradientFD, TestIotasJAXResolveFD, TestBoozerResidualAdjointFD, TestNonQSRatioJAXResolveFD) are real but use loose acceptance: `rel < 1e-3 OR abs < 1e-8`, and only require 2/3 of probed directions to validate.
4. The "CPU reference" file `test_single_stage_jax_cpu_reference.py` does run the genuine C++ (`simsoptpp`) solvers and compares them. However, **the same-state wrapper-value parity tests use the JAX surface for both legs** and only the LS-warmed `boozer_setup` independently runs CPU and JAX through `simsoptpp` and `BoozerSurfaceJAX` respectively. CPU reference is genuine.
5. Adjoint-failure-→-NaN-gradient contract is well-covered by 6 dedicated tests but only at the compiled-bundle/host-wrapper boundary (lines 286, 2841, 3153, 3203, 916, 938). Nothing exercises the legacy `BoozerResidualJAX.dJ()` / `IotasJAX.dJ()` / `NonQuasiSymmetricRatioJAX.dJ()` public-Optimizable failure path with a real failed solve, only with monkeypatched callbacks.

## 3. Missing coverage (P0 gaps)

* **End-to-end IFT-vs-FD on the real reduced fixture for IotasJAX** at multiple `iota_target_shift` values without re-solve (i.e., perturb `iota_target` only, keep the LS fixture fixed) so the inner-state rerun is deterministic. Suggested tolerance: `rtol=1e-6` directional, `eps=1e-6` central FD.
* **`exact-well-conditioned-adjoint` lane in this bucket**: `test_iotas_jax_exact_well_conditioned_gradient_matches_dense_projection` (line 1728 in `test_surface_objectives_jax.py`) is a toy 3x3 self-test; the production gate must consume an actual `BoozerSurfaceJAX(boozer_type='exact')` fixture, materialize `J_dense = scipy.linalg.lu` from the operator-applied basis, and assert `np.testing.assert_allclose(operator_solve, lu_solve(J_dense, rhs), rtol=1e-6, atol=1e-8)` plus `residual_rel <= 1e-10`. The plan acknowledges this lane lives elsewhere, but if the M5 file claims to validate it then the gate must run here too.
* **Successful forward + failed adjoint → non-finite gradient** for the legacy `BoozerResidualJAX.dJ()` / `IotasJAX.dJ()` / `NonQuasiSymmetricRatioJAX.dJ()` (NOT the traceable bundle): construct a real BoozerSurfaceJAX where `res["success"]=True` but a programmatic poison on `solve_transpose_with_status` returns `(garbage, False)`. Today only `bad_res["success"]=False` (line 5247) is tested, which short-circuits before the adjoint solve. Suggested tolerance: `assert not np.all(np.isfinite(grad))`.
* **CPU-vs-JAX projected coil-derivative parity** at the LS-warmed fixture for IotasJAX/NonQS (currently only BoozerResidual same-state value parity is tight; gradients drop to `_LS_WRAPPER_GRADIENT_TOLS` which has loose default rtol — verify and tighten).
* **Branch-stable composite descent step**: `TestCompositeGradientPipeline.test_branch_stable_composite_gradient_fd_and_descent` is the only descent-step gate. Replicate for the standalone `BoozerResidualJAX` and `NonQuasiSymmetricRatioJAX` so a sign error in the IFT term is detectable in isolation.
* **Per-axis FD on `NonQuasiSymmetricRatioJAX.dJ()`**: the only QS-ratio FD validation requires `_REAL_RESOLVE_FD_MIN_STABLE_SAMPLES=3` of 5 directions and accepts Taylor rate 0.55. A coordinate-axis sweep on the controlled LS fixture with `rtol=1e-4` per direction is missing.
* **Cross-CPU parity for `NonQuasiSymmetricRatioJAX.dJ()`**: there is per-component value parity (lines 4731, 5613–5630) but the same-state gradient parity reuses `_LS_WRAPPER_GRADIENT_TOLS` which is wide. Add a strict `rtol=1e-6` gradient parity gate at the same LS-converged state.
* **GPU same-state determinism**: `TestRealFixtureGpuM5Parity` runs the GPU lane `_GPU_RUNTIME_TOLS["same_state_gradient_rtol"]` but does not pin reproducibility (run twice, same output). Add a determinism gate with `np.testing.assert_array_equal`.
* **`make_traceable_objective_runtime_bundle` cache key invalidation on `outer_objective_config["vessel_gamma"]` value change** (not just type/shape) — currently only `iota_target` and `solver_generation` invalidation is tested.

## 4. Tightening playbook (P0)

1. **Strip the "or abs_err < 1e-8" escape hatch** in TestBoozerResidualGradientFD.test_end_to_end_dJ_vs_fd (line 4213). Use `np.testing.assert_allclose(dd_composed, dd_fd, rtol=_FD_GRADIENT_TOLS["directional_fd_rtol"], atol=_FD_GRADIENT_TOLS["directional_fd_atol"])`.
2. **Require ALL stable directions to validate** in `_assert_wrapper_resolve_fd_matches_real_fixture` (line 1990) and TestIotasJAXResolveFD (line 5804). Replace `validated_directions >= 2` and `_REAL_RESOLVE_FD_MIN_STABLE_SAMPLES = 3` with `len(stable_samples) == len(direction_samples)` after filtering branch-switches. The 2-of-3 / 3-of-5 majority gate hides systematic adjoint bugs.
3. **Drop the Taylor rate from 0.55 to 0.4** (`_REAL_RESOLVE_FD_TAYLOR_RATE`, line 1439). Symmetric central FD with re-solve gives second-order convergence (≈0.25); 0.55 accepts non-decreasing FD error.
4. **Promote the adjoint-fraction "diagnostic" to a real gate** (line 5917). Either `assert adjoint_fraction > 0.05` so the IFT term provably contributes, or remove the test and keep the diagnostic as a plain logger inside the resolve-FD test. Today it is dead weight.

## 6. CPU-reference audit (`test_single_stage_jax_cpu_reference.py`)

**Verdict: the CPU reference is genuine, with one caveat.**

* The fixture `_make_boozer_setup` (line 2051) builds two **independent** stacks: a `simsoptpp`-backed `BiotSavart`+`BoozerSurface` and a JAX-backed `BiotSavartJAX`+`BoozerSurfaceJAX`. Both run `run_code(iota0, G0)` from the same scalar initial guess; neither legs reads from the other. This is a real CPU oracle.
* `_cpu_single_stage_wrapper_values()` and `_cpu_single_stage_wrapper_gradients()` (lines 522, 530) construct `BoozerResidual`, `Iotas`, `NonQuasiSymmetricRatio` from `simsopt.geo.surfaceobjectives`, NOT from `simsopt.geo.surfaceobjectives_jax`. So the CPU value/grad path runs through C++ and SciPy, not through JAX shims.
* In `test_real_fixture_ondevice_parity_and_wrapper_gradients` (line 4687) the comparison is built from `_build_real_fixture_ondevice_m5_same_state_pair()`. Reading the comments and code, "same_state" means the JAX state (DOFs, iota, G) is **copied into** the CPU fixture so that any divergence is purely value-path implementation. This is legitimate `ls-wrapper-gradient` lane coverage as long as the user remembers it is *not* an end-to-end CPU run.
* In `TestRealFixtureGpuM5Parity::test_real_fixture_gpu_wrapper_values_and_gradients_match_cpu_reference` (line 4810) the CPU result is captured first, then the GPU fixture is built **with the CPU surface DOFs and iota/G** as overrides. So the GPU is comparing against the CPU-converged state, which is again a value/grad path test, NOT a path-equality test. Properly labelled.
* **The one caveat**: `TestExactSolveCPUJAXParity` (line 5585) uses `_solve_exact_cpu_jax_parity_pair()` (line 5460) to build the pair. I did not exhaustively read that helper, so verify on a final pass whether both the CPU and JAX exact solves run independently from the LS warm-start, or whether one is seeded from the other. The class docstring claims "Both solvers start from the same LS-warmed initial guess (shared surface DOFs, iota, G)" — if so, the parity is genuine; if one is seeded from the other's exact solution that would be a CPU-reference overmatch.

No JAX shadow of CPU exists in this file. The "CPU reference" stack is the real `simsoptpp`-backed stack, run independently.
