## Scope

Prompt item 12 originally covers four pure analytic fields from
`src/simsopt/field/magneticfieldclasses.py`:

- `ToroidalField` — `B = B0*R0/R * e_phi` with full B/A/dB/dA/d2B/d2A.
- `PoloidalField` — `B = (B0/(R0*q)) * r * e_theta` with B and dB only.
- `MirrorModel` — WHAM double-Lorentzian flux function with B and dB.
- `CircularCoil` — off-axis circular-coil field via complete elliptic
  integrals `K(k^2)`, `E(k^2)`. **DEFERRED** because
  `jax.scipy.special.ellipk` / `ellipe` are not exposed by `jaxlib` 0.10.0.

`ScalarPotentialRZMagneticField` is owned by P3 item 23 and is not in this
item's scope.

This run closes the first three. The `CircularCoil` sub-item is recorded as
the only outstanding piece of item 12 with a concrete resolution path
(implement a JAX-native Carlson `R_F`/`R_D` helper or Bulirsch `cel`, then
build `CircularCoil` on top of that helper).

## Risk Register

- Upstream layout inconsistency: `ToroidalField` / `PoloidalField` store
  `dB[p, l, j]` (component, derivative) whereas `MirrorModel` stores
  `dB[p, j, l]` (derivative, component). The JAX kernels match each class
  exactly. Mitigation: documented in `12-invariants.md` and in module
  docstring; parity tests cover both layouts.
- Upstream typo: `ToroidalField._d2B_by_dXdX_impl` has the `3*x^2 + y^3`
  entry that does not match the analytic third derivative of `_B_impl`.
  The JAX kernel reproduces the upstream literal expression so the
  parity gate matches the CPU oracle bit-for-bit. The downstream parity
  contract is "match the CPU class", not "match the textbook analytic
  expression"; we record the deviation here so a future bug-fix in
  upstream gets a coordinated update in the JAX kernel.
- Singular regimes: `PoloidalField` at `R = R0`, `MirrorModel` at
  `R = 0`. The CPU class produces `NaN`/`inf`; the JAX kernel matches.
  Parity tests exclude those regions to keep `assert_allclose` clean.
