## Scope

Prompt item 13 covers the JAX port of the C++ regular-grid interpolant
kernel:

- C++ source: `src/simsoptpp/regular_grid_interpolant_3d.h`,
  `src/simsoptpp/regular_grid_interpolant_3d_impl.h`,
  `src/simsoptpp/regular_grid_interpolant_3d_c.cpp`,
  `src/simsoptpp/regular_grid_interpolant_3d_py.cpp`.
- New JAX module: `src/simsopt/jax_core/regular_grid_interp.py`.

The new module re-implements the C++ kernel as a JAX-compatible
piecewise polynomial interpolant on a rectangular cuboid mesh with
tensor-product 1D Lagrange polynomials. The build phase uses pure
NumPy (linspace mesh, skip-mask, DOF evaluation); the evaluate phase is
JAX-jittable and consumes immutable `RegularGridInterpolant3DSpec`
state. Downstream wrappers in `field/magneticfieldclasses.py`
(`InterpolatedField`), `field/tracing.py`, and `geo/surface.py`
continue to use the C++ binding. Item 15 closes those wrapper
consumers.

## Source Audit

- `RegularGridInterpolant3D::evaluate_inplace` performs the soft
  boundary clamp at `regular_grid_interpolant_3d_impl.h:86-95` with
  `_EPS_ = 1e-13`, then computes the cell index and local fractional
  coordinate, then calls `evaluate_local`.
- `RegularGridInterpolant3D::evaluate_local` looks up the
  per-cell value table from `all_local_vals_map`. Skipped or OOB cells
  hit the `out_of_bounds_ok` branch at line 119 and leave the caller's
  output buffer unchanged.
- `RegularGridInterpolant3D::evaluate_local` triple-loop is the
  O(degree^4) hot kernel; it FMAs `pkxs[i] * pkys[j] * pkzs[k] *
  vals_local`.
- `UniformInterpolationRule` and `ChebyshevInterpolationRule` build the
  Lagrange node sets at `regular_grid_interpolant_3d.h:312-343`.
- `interpolate_batch` evaluates the user function on the retained DOF
  set in batches of 16384 and packs the resulting values into a
  per-cell table at `regular_grid_interpolant_3d_impl.h:14-56`.

## Implementation Changes

- `src/simsopt/jax_core/regular_grid_interp.py` adds:
  - Frozen `InterpolationRule`, `UniformInterpolationRule`, and
    `ChebyshevInterpolationRule` constructors.
  - Frozen `RegularGridInterpolant3DSpec` with explicit mesh metadata
    and a per-cell `cell_table` of shape `(cells_to_keep + 1,
    degree+1, degree+1, degree+1, value_size)`. The trailing zero
    sentinel row is the destination of every skipped or OOB query.
  - `build_regular_grid_interpolant_3d` builds the cell table from a
    user function in pure NumPy; this matches `interpolate_batch` in
    the C++ kernel.
  - `evaluate_batch` is `jit`-compiled with `degree`, `value_size`,
    and `out_of_bounds_ok` as static arguments. The per-sample
    evaluator is vmapped over the leading axis of `xyz`. The basis
    contraction uses `jnp.einsum("i,j,k,ijkl->l", pkx, pky, pkz,
    local_vals)`.
  - `estimate_error` mirrors the C++ random-sample error estimator.
- `tests/jax_core/__init__.py` is added so the new test directory is
  picked up by pytest.
- `tests/jax_core/test_regular_grid_interp_item13.py` adds 28
  parametrized parity tests covering polynomial exactness,
  uniform/Chebyshev node parity, OOB behavior, skip semantics, C++
  cross-oracle parity, skip-mask cross-oracle parity, the
  `estimate_error` helper, and `jit`-traceability.

The C++ binding `simsoptpp.RegularGridInterpolant3D` is left
unchanged and remains the parity oracle. Public field classes that
still consume the C++ binding (`InterpolatedField`,
`RegularGridInterpolant3D` classifier, surface-distance interpolant)
continue to work; they are addressed in items 15 / 16 and the surface
distance follow-up.

## Bench Artifact

See `.artifacts/jax_port_goal/bench/13.json`. CPU-only bench at
`16 × 16 × 16` cells, degree 4, value size 3, 512 evaluation points,
100 timed calls after 5 warmups. The JAX median per-batch time is
`6.30e-04 s`; the C++ oracle median is `9.05e-05 s`. The JAX kernel is
roughly 7x slower than the SIMD-FMA C++ loop on this representative
fixture but is fully `jit`-compatible and lives downstream of the
generic XLA compiler optimisations. The benchmark is reporting-only:
there is no perf-regression gate on this item because it is a new
kernel port with no existing JAX baseline.
