# Item 12 — Serialization / Restart Compatibility

## Public API surface affected

This item adds a new pure-JAX kernel module
`src/simsopt/jax_core/analytic_pure_fields.py` and three immutable spec
dataclasses. It does **not** modify any public Python classes:

- `simsopt.field.magneticfieldclasses.ToroidalField` — unchanged.
- `simsopt.field.magneticfieldclasses.PoloidalField` — unchanged.
- `simsopt.field.magneticfieldclasses.MirrorModel` — unchanged.

The legacy `MagneticField` wrappers (`set_points`, `B()`, `dB_by_dX()`,
…) and their `as_dict` / `from_dict` serializers continue to use the
existing CPU `_*_impl` paths and the existing `from_dict` constructors.
JAX kernels are accessed only by callers that explicitly construct one
of the new spec dataclasses.

## Serialization round-trip

- `ToroidalField.as_dict` / `from_dict` (`magneticfieldclasses.py:114-125`)
  remain unchanged.
- `PoloidalField.as_dict` / `from_dict` (`magneticfieldclasses.py:222-233`)
  remain unchanged.
- `MirrorModel.as_dict` / `from_dict` (`magneticfieldclasses.py:1016-1027`)
  remain unchanged.

The CPU oracle test
`tests/field/test_magneticfields.py::Testing::test_toroidal_field` exercises
the SIMSON round-trip end-to-end on the unmodified CPU class; this run did
not touch that test or that class, so the existing serialization contract
is preserved.

## Restart JSON

There is no restart-JSON consumer for these analytic fields in the active
manifest. P2 item 15 will own the JAX-backed wrapper that consumes these
new kernels; serialization there will be designed around `ToroidalFieldSpec`
/ `PoloidalFieldSpec` / `MirrorModelSpec` if needed.

## Compatibility metadata

- `ToroidalFieldSpec`, `PoloidalFieldSpec`, `MirrorModelSpec` are
  frozen dataclasses registered as JAX pytrees with their float fields
  declared as `meta_fields`. They are hashable and equality-stable
  under `dataclasses.replace`. They are not part of any existing
  `from_dict` consumer, so no migration path is needed.
- The JAX kernel module is imported lazily by callers; the package-level
  `simsopt.jax_core` `__init__.py` does **not** re-export
  `analytic_pure_fields` symbols, so any CPU-only install path remains
  unaffected. (P2 item 15 may decide to expose them through the package
  surface when the JAX-backed wrappers land.)
